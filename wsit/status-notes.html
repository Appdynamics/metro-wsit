WSIT Status Notes

Date: May 13, 2006

Introduction

This document provides the status of the interoperability testing performed by Sun Microsystems on each of the technologies supported in Web Services Interoperability Technology (WSIT). It also lists and describes any known issues.

In conjunction with Microsoft, Sun has done limited interoperability testing to ensure that the Web services created using WSIT technologies can access and consume Web services created using Microsoft's WCF product and vice versa.

Glassfish Classpath Change Requirements

The b43-nightly-01_may_2006 build of GlassFish contains several Jar files which contain classes that can conflict with the newer versions of those classes provided by the WSIT environment. To avoid problems, the classpath for both the Glassfish application server and the client application must be modified.

The WSIT installer (wsit-on-glassfish.xml) will handle making the necessary classpath modifications in the Glassfish configuration file to prevent problems. However, the client applications need to have equivalent changes. The samples accompanying the Getting Started with WSIT How To documents already have the necessary work-around in place. The work-around is simply to exclude the conflicting Jar files from the application's classpath. Should you create a build.xml file of your own, you will need to do the same thing. The following code snippet excludes the conflicting Jar files from the classpath used to run the tools and the client application:

<path id="wsit.classpath">
<pathelement location=
"${java.home}/../lib/tools.jar"/>
<fileset dir="${lib.home}">
<include name="*.jar"/>
<exclude name="jaxws-update.jar"/>
<exclude name="j2ee*.jar"/>
<exclude name="javaee.jar"/>
<exclude name="appserv-*.jar"/>
</fileset>
</path>

What is Not Implemented

All Metadata Exchange functionality has been implemented, including send and receive metadata and the client API for other components to use. However, the code that adds headers to the response message is commented out because of a known issue in xml stream buffer.

Known Issues

Interoperability Feature

Status/Workaround

Support for returning WS-Addressing response headers with response messages.

This is an unbound prefix issue. This issue is preventing the Metadata Exchange feature from returning the WS-Addressing response headers that are required by the Metadata Exchange specification.

WORKAROUND: The client code currently does not check for these headers so the response is still parsed.

Support for multiple Web services on a single server

If more than one Web service is deployed in the server, a client can only use Metadata Exchange to retrieve the first Web service deployed.

WORKAROUND: Do not use Metadata Exchange. Instead, use the wsimport tool and specify the "?wsdl" address as the target.

Support for ReplyTo in client request.

This feature is currently not supported.

Interoperability Feature

Status/Workaround

Using MTOM with SOAP 1.2

MTOM with SOAP 1.2 will not work if the SOAP message to be sent has no soap action, that is, it will not work in either of the following cases:

- the WSDL has no soap action

- WS-Addressing is disabled

WORKAROUND: If the WSDL has a SOAP action or WS-Addressing is enabled, then MTOM will work.

What is Not Implemented

The following features have been implemented but are not working correctly:

  • Multiple secure conversation tokens per message is not implemented. Only one secure conversation token per message is assumed.
  • Session timeout handling for secure conversation sessions and LifeTime handling for IssuedTokens.
  • MaxNonceAge information item specified in NetBeans IDE user interface and configuration files WSDL will not be honored, default values will be used instead.
  • <sp:SAMLToken> scenarios untested. SAML was only tested on the WS-Trust integration path.
  • If a binary secret arrives in a SAML assertion, then ensure SSL is used
  • Policy Verification is not done in the case of Encrypted Body scenarios.
  • Policy Verification for WS-Trust Protocol Messages.
  • Backward Compatibility with XWSS 2.0 FCS.
  • Implied DerivedKeys specified by a @wsc:nonce attribute on SecureTokenRequest
  • The @wsc:Instance attribute on wsse:Reference for referencing specific Secure Context Token instances.
  • Reference Type verification in Policy Verification.
  • Logging code added for WSIT is incomplete.
  • The sha256 signature algorithm is not supported and therefore all AlgorithmSuite identifiers that imply the sha256 signature algorithm are not supported.
  • Some of the JAX-WS with WSIT extensions that works without extensions is not enabled. For example, Handlers and MustUnderstand Processing are not enabled in the WSIT extensions bundle because a different PipelineAssembler is used.

NetBeans WSIT Module Status

What is Not Implemented

  • WSIT support for Web Services and Web service clients is implemented in Web projects only.
  • Web Service Tester functionality for WSIT enabled Web services is not implemented.

Known Issues

Interoperability Feature

Status/Workaround

Client-side classpath must not use the jaxws-update.jar

For the client side classpath, make sure that jaxws-update.jar is NOT be in the classpath. You can do this by spelling out each jar name in the classpath (this is the best way to avoid confusion) or you can include all jars and exclude jaxws-update.jar from the client classpath.

WSIT Security: Alpha input can be saved to numeric Timestamp Freshness Limit.

EXPECTED BEHAVIOR: Timestamp Freshness Limit in Security -> Advanced Configuration should only accept numerics.

ACTUAL BEHAVIOR: Accepts and saves alphas to the WSDL.

WSIT Security: Alpha input can be saved to Maximum Clock.

EXPECTED BEHAVIOR: Maximum Clock Skew in Security -> Advanced Configuration should only accept numerics.

ACTUAL BEHAVIOR: Accepts and saves alphas to the WSDL.

WSIT Security: A value that exceeds the maximum allowed can be saved to the Max Nonce Age.

EXPECTED BEHAVIOR: Max Nonce Age in Security -> Advanced Configuration should not accept a value greater than 900000.

ACTUAL BEHAVIOR: Accepts and saves invalid value to the WSDL.

WSIT Security: A value that exceeds the maximum allowed can be saved to the Max Clock Skew.

EXPECTED BEHAVIOR: Max Nonce Age in Security -> Advanced Configuration should not accept a value greater than 1200000.

ACTUAL BEHAVIOR: Accepts and saves invalid value (1200001) to the WSDL.

WSIT Security: Include Token in a message and X509 Version can be set incorrectly.

EXPECTED BEHAVIOR: Saved details show Security Binding and Encryption token with the same values.

ACTUAL BEHAVIOR: Saved values are incorrect.

WSIT Security Keystore Configuration: Behavior of child dialogs is incorrect.

EXPECTED BEHAVIOR: Do not expect values that specified on child dialogs to be saved to the WSDL.

ACTUAL BEHAVIOR: Values that are specified in child dialogs are saved to the WSDL. Saving should not occur until the OK button is clicked on parent dialog.

WSIT Accessibility: ALT + letter access for command buttons is not working correctly.

None of the command buttons in the Tooling plug-in provide for access using ALT + letter.

WSIT General: Attributes are saved to wsit.xml even when user cancels.

EXPECTED BEHAVIOR: If the user presses Cancel, then none of the currently changed values should be saved to wsit.xml.

ACTUAL BEHAVIOR: When attributes are changed in Edit Web Service Attributes window, the wsit.xml shows the attributes as being updated before the OK button is clicked. If the user chooses to Cancel the changes, the changed attributes are still saved to the file.

WSIT: Limit on size of MaxBufferSize and Inactivity Timeout for Reliable Messaging.

EXPECTED BEHAVIOR: User should not be allowed to enter more than eight (8) digits in either of the text boxes.

ACTUAL BEHAVIOR: User is allowed to enter more than eight digits are entered, however, an error is received on the status line.
